## 0. 架构决策记录（ADR）

> 本节集中记录在脑暴阶段已锁定的关键设计决策，是后续所有功能规格的"公理层"，不允许在迭代中随意推翻。

| 编号 | 维度 | 决策 | 理由 |
|------|------|------|------|
| ADR-01 | 输入精度 | 城市名模糊匹配 + 自动 geocoding，不暴露经纬度给用户 | 降低用户心智负担，精度由后端保证 |
| ADR-02 | 时区/DST | 接入 Google Timezone API，构建本地历史 DST 规则补丁库 | 第三方实时 API 覆盖当前时区，历史 DST 需要本地修正（尤其中国 1986–1991）|
| ADR-03 | 生时置信度 | 三档：精确（±5min）/ 大约（±1h）/ 不清楚。不清楚时默认 **12:00 正午** | 正午出生上升点误差最小，是业界通行 fallback |
| ADR-04 | 分宫制 | MVP 支持 **Placidus（默认）+ Whole Sign（整宫制）**，Koch / Equal 列入 v1.1 | 两者覆盖 90% 用户需求，避免过早膨胀配置项 |
| ADR-05 | 星体范围 | 十大行星 + ASC/MC + 北交点/南交点 + 凯龙星 + 黑月莉莉丝（True Lilith）；恒星计算列入 v2.0 | 凯龙/莉莉丝在华人占星社区接受度高；恒星工程量大留后期 |
| ADR-06 | 容许度 | 系统内置默认值，用户可在「高级设置」覆盖，范围 1°–15°；生时不准时全局收紧 2° | 默认值保护新手，开放性满足专业用户 |
| ADR-07 | 推运深度 | MVP 仅做 **Transit（行运/流年）**；Secondary Progressions 和 Solar Arc 列入 v1.5 | Transit 是最高频需求，先跑通闭环 |
| ADR-08 | 合盘逻辑 | **相位计分制** 输出匹配度（0–100），并附分项文字说明 | 数字化结果可分享、可截图，强社交属性 |
| ADR-09 | 绘图技术 | 移动端使用 **SVG** 实现可缩放可交互星盘；后备导出用 Canvas 转 PNG | SVG 支持 CSS 动画，适合点击高亮交互 |
| ADR-10 | 内容解析 | **混合架构**：结构化 JSON 库（原子级查表）+ LLM Prompt 生成段落级综合报告 | 查表保证准确性和速度，LLM 负责语感和个性化 |
| ADR-11 | 报告导出 | 支持**长图分享**（PNG）和 **PDF 下载**，两者均含水印品牌标识 | 高社交传播价值，同时保护品牌曝光 |

## 1. 产品概述

### 1.1 产品定位

一款面向 Z 世代占星爱好者的**专业级本命盘 + 推运分析**移动端应用。核心差异点：

- **计算级专业**：天文精度对标桌面端专业软件（Astro.com 同源算法）
- **交互级简洁**：移动端原生体验，无学习成本
- **解析级智能**：LLM 驱动的个性化报告，非千人一面的查表文本

### 1.2 目标用户

| 用户层 | 特征 | 核心诉求 |
|--------|------|----------|
| 探索型（60%）| 18–28 岁，初级爱好者 | 快速获得有趣且准确的性格洞察 |
| 研究型（30%）| 有一定占星基础，会手动盘 | 需要专业计算工具替代桌面软件 |
| 专业型（10%）| 占星师、KOL | 需要客户报告生成和批量盘功能 |

### 1.3 MVP 范围（本文档覆盖）

```
本命盘计算 → 交互式星盘 SVG → 分层解读报告 → 行运推算（Transit）→ 合盘匹配 → 报告导出
```

## 2. 功能规格

### 2.1 模块一：出生数据输入

#### 2.1.1 功能描述

用户输入出生信息，系统自动完成地理编码、时区解析和 DST 修正，返回高精度计算参数。

#### 2.1.2 输入字段

| 字段 | 输入方式 | 校验规则 | 说明 |
|------|----------|----------|------|
| 姓名/昵称 | 文本 | 1–20 字符 | 仅用于展示 |
| 出生日期 | 日期选择器 | 1900–2025 | 不支持未来日期 |
| 出生时间 | 时间选择器 | HH:MM | 见时间置信度 |
| 出生地 | 搜索输入框 | 非空 | 自动补全城市名 |
| 时间置信度 | 单选三档 | 必选 | 见下方说明 |

#### 2.1.3 时间置信度三档

```
● 精确（±5分钟）   → 正常计算，完整展示宫头精度
○ 大约（±1小时）   → 全局容许度收紧 2°，上升/中天展示为区间（±3°）
○ 不清楚           → 默认 12:00 正午，隐藏上升/宫位解读，仅展示行星/星座
```

#### 2.1.4 地理编码流程

```
用户输入城市名
    ↓
前端：调用高德 Places API（中国 IP）/ Google Places API（海外 IP）
    ↓
返回候选列表（最多 5 条），用户确认（含国家、省份消歧义）
    ↓
获取精确经纬度（精度：小数点后 6 位）→ 存入 birth_data
    ↓
后端：调用 Google Timezone API 获取 UTC 偏移 + DST 标志
    ↓
DST 补丁层：检查历史 DST 规则库，自动修正 UTC 偏移
    ↓
返回最终 UTC 时间戳供计算引擎使用
```

#### 2.1.5 历史 DST 补丁库（必须内置）

| 国家/地区 | DST 时段 | 偏移修正 |
|-----------|----------|----------|
| 中国 | 1986-04 ~ 1991-09 | +1h |
| 台湾 | 1945 ~ 1979（断续）| +1h |
| 香港 | 1941 ~ 1979（断续）| +1h |
| 苏联/俄罗斯 | 历年规则 | 参照 IANA tzdata |
| 其他地区 | 完整 IANA tzdata | 动态加载 |

> 工程实现：内置 `tzdata` npm 包 + 针对中华地区的自定义 patch JSON。

### 2.2 模块二：天文计算引擎

#### 2.2.1 计算内核

- **服务端**：Python + `pyswisseph`（Swiss Ephemeris，精度 J2000.0 起 5000 年）
- **星历数据**：SE1 格式文件，存储于 OSS，按需懒加载
- **API 形式**：REST `/v1/chart/calculate`，POST，返回 JSON

#### 2.2.2 计算参数配置

**分宫制（House System）**

| 代码 | 名称 | 默认 | 适用场景 |
|------|------|------|----------|
| `P` | Placidus 普拉西度 | ✅ | 现代西占主流 |
| `W` | Whole Sign 整宫制 | 可选 | 古典/赫勒尼克占星 |

> 切换入口：「星盘详情页」右上角设置图标 → 分宫制

**计算星体清单（MVP）**

```
行星（10）：☉太阳 ☽月亮 ☿水星 ♀金星 ♂火星 ♃木星 ♄土星 ♅天王星 ♆海王星 ♇冥王星
轴点（2）：ASC 上升点、MC 中天
交点（2）：☊北交点（True Node）、☋南交点
小天体（2）：⚷凯龙星、⚸黑月莉莉丝（True Lilith）
虚点（1）：⊕福点（Pars Fortunae）
```

**相位类型与默认容许度**

| 相位 | 角度 | 默认容许度 | 用户可调范围 |
|------|------|-----------|-------------|
| 合相 Conjunction | 0° | 8° | 4°–12° |
| 对分相 Opposition | 180° | 8° | 4°–12° |
| 三分相 Trine | 120° | 7° | 3°–10° |
| 四分相 Square | 90° | 7° | 3°–10° |
| 六分相 Sextile | 60° | 5° | 2°–8° |
| 半六分相 Semi-Sextile | 30° | 2° | 1°–4° |
| 十二分相 Quincunx | 150° | 3° | 1°–5° |

> 生时不准时，所有容许度在用户设置值基础上额外 **-2°**（最小不低于 1°）。

### 2.3 模块三：交互式星盘 SVG

#### 2.3.1 星盘结构

```
外圈：黄道十二宫（30° × 12 = 360°）

内圈：行星符号散布（按黄道经度定位）
中心区：相位线（aspect lines）
```

#### 2.3.2 交互行为

**点击行星**
1. 对应行星符号放大 1.3x，高亮为主色调
2. 与该行星有相位的连线**高亮**（按相位类型着色）

**双指缩放**
- SVG viewport 支持缩放（0.8x–3x）
- 符号字号自适应（不随缩放变形）

**长按星盘空白区**
- 触发"分享快照"：生成当前视角的 PNG

### 2.4 模块四：内容解析引擎

#### 2.4.1 双层架构

```
第一层（原子库）：JSON KV 数据库
    ├── 行星 × 星座（10 × 12 = 120 条）
    ├── 行星 × 宫位（10 × 12 = 120 条）
    └── 行星 × 相位 × 行星（~300 核心条目）

第二层（LLM 合成）：Prompt Engineering
    输入：第一层原子特征 + 权重算法输出的特征标签向量
    输出：结构化个性化报告段落
    模型：Claude claude-sonnet-4-20250514（API 调用，服务端）
```

#### 2.4.2 特征权重算法

以下配置被视为"主导特征"，系统自动打标签：

| 规则 | 触发条件 | 标签示例 |
|------|----------|----------|
| 群星星座 | ≥3 颗行星落同一星座 | `#摩羯重仓` |
| 宫位聚集 | ≥3 颗行星落同一宫位 | `#第七宫强调` |
| 元素失衡 | 某元素行星数 ≤1 | `#缺水象` |
| 行星强势 | 行星落入尊贵位（庙/旺） | `#金星在天秤庙位` |
| 行星弱势 | 行星落入劣势位（陷/落） | `#土星在白羊落位` |
| 逆行群 | ≥4 颗行星同时逆行 | `#大量逆行` |
| 图形相位 | 大三角/十字等图形 | `#风象大三角` |

这些标签作为 Prompt 的上下文注入，影响 LLM 报告的叙述重心。

#### 2.4.3 报告分层结构

```
【报告首页】摘要卡片（≤200字，核心特质 Top3）
    ↓
【模块一】核心人格（太阳/月亮/上升三角）
    ↓
【模块二】情感与关系模式（金星/火星/第7宫）
    ↓
【模块三】职业与天赋（第10宫/土星/北交点）
    ↓
【模块四】心理底层模式（月亮/冥王星/凯龙）
    ↓
【模块五】今年行运简报（Transit，最近3个月关键行运）
```

每个模块独立加载（懒渲染），防止首屏等待过长。

#### 2.4.4 LLM 调用规格

```
System Prompt（固定层）:
你是一位融合现代心理学的专业占星师，语言风格：温暖、洞察、不神秘化。
输出格式：纯中文，每段 80–150 字，不使用 Markdown 标题。

User Prompt（动态层）:
[用户昵称]的本命盘原子特征：
- 太阳狮子10宫：{json原子文案}
- 月亮金牛5宫：{json原子文案}
- 主导标签：#摩羯重仓 #缺水象
- 关键相位：太阳三分木星（容许度2.1°）、月亮四分土星（容许度1.8°）

请生成「核心人格」模块的解读，重点体现标签的叙事权重。
```

#### 2.4.5 缓存策略

- 同一出生数据 + 同一报告模块：**永久缓存**（存储于用户账户云端）
- 行运报告（Transit）：**每日失效**（依赖当日行星位置）
- 首次生成耗时：≤8s；命中缓存：≤500ms

### 2.5 模块五：行运推算（Transit）

#### 2.5.1 功能描述

计算当前（或指定日期）行星位置与本命盘行星的实时相位，生成行运日历和运势波动曲线。

#### 2.5.2 运势波动曲线图

- X 轴：时间（默认展示未来 90 天，可拖拽滑动到过去/未来 1 年）
- Y 轴：综合运势分（0–100，由相位强度加权计算）
- 曲线分层：总分线（粗）+ 情感线 / 事业线 / 能量线（细，可切换显示）
- **关键事件标注**：在曲线上标记重要相位（行星换座、逆行起止、日月食）

**波动分值计算公式：**

```
daily_score = Σ(aspect_weight × planet_weight × (1 - orb/max_orb))

aspect_weight:
  conjunction=1.0, opposition=0.9, square=0.8
  trine=0.7, sextile=0.5

planet_weight:
  SUN/MOO=1.0, JUP/SAT=0.9, VEN/MAR=0.7
  URA/NEP/PLU=0.5（慢速行星影响均匀，加衰减）

分值归一化至 0–100，70+ 为利好，40– 为需注意
```

#### 2.5.3 行运日历视图

- 月历格式，每天显示颜色等级（绿/黄/红）
- 点击日期显示当天详情：当日行运相位列表 + 简短文字说明

### 2.6 模块六：合盘（Synastry）

#### 2.6.1 功能描述

上传两人本命盘数据，计算星盘互动相位，输出匹配度评分和分项分析。

#### 2.6.2 匹配度计算

**总分 = 加权平均，满分 100**

| 维度 | 权重 | 计算逻辑 |
|------|------|----------|
| 情感共鸣 | 35% | 金星、月亮的相互相位 |
| 吸引力 | 25% | 火星、金星、上升的跨盘相位 |
| 沟通匹配 | 20% | 水星、上升的相位兼容性 |
| 长期稳定 | 20% | 土星与对方个人行星的相位 |

- 和谐相位（三分/六分/合相）加分
- 张力相位（四分/对分）扣分（但附注"有助成长"说明）

#### 2.6.3 合盘双圈图

- 内圈：自己的星盘
- 外圈：对方的星盘
- 跨盘相位线：单独颜色标注
- 支持切换查看"单盘"和"双圈叠加"视图

### 2.7 模块七：报告导出

#### 2.7.1 长图导出（PNG）

- 包含：用户昵称、出生信息摘要、星盘缩略图、完整报告文字
- 尺寸：宽 750px，高度自适应（约 3000–6000px）
- 水印：App Logo + 网站地址，透明度 20%，位于底部

#### 2.7.2 PDF 导出

- 标准 A4，2 列排版
- 含页眉（App Logo）、页脚（页码 + 免责声明）
- 支持"分享给占星师"的专业版式选项
- 生成耗时：≤10s

#### 2.7.3 免责声明（必须展示）

## 3. 数据模型

### 3.1 用户档案

```typescript
interface UserProfile {
  id: string;                    // UUID
  nickname: string;
  birth_data: BirthData;
  created_at: string;            // ISO 8601
  house_system_pref: "P" | "W";
  orb_overrides?: Record<string, number>;
  saved_charts: SavedChart[];
}

interface BirthData {
  datetime_utc: string;          // ISO 8601，UTC
  datetime_local: string;        // 用户本地时间（展示用）
  timezone_id: string;           // 如 "Asia/Shanghai"
  dst_applied: boolean;
  dst_offset_minutes: number;
  city_display: string;          // 如 "上海, 中国"
  latitude: number;              // 精度 6 位小数
  longitude: number;
  time_confidence: "exact" | "approximate" | "unknown";
}
```

### 3.2 星盘计算结果

## 6. 非功能性需求

### 6.1 数据安全

- 出生数据属于个人敏感信息，传输全程 HTTPS，静态加密存储（AES-256）
- 用户可一键删除所有数据（GDPR / 中国个保法合规）
- LLM 调用时不传输用户真实姓名，仅传昵称和星盘特征

### 6.2 免责声明展示要求

以下场景**强制**展示免责声明：
1. 首次使用 App 时（确认弹窗）
2. 每份报告底部
3. 导出的 PNG/PDF 中

### 6.3 无障碍

- 星盘图必须提供**文字替代描述**（行星落位列表即可作为 a11y 替代）
- 颜色方案通过 WCAG AA 对比度标准
