## 1. 产品概述

「春风-塔罗」是「春风-奥秘」模块下的核心占卜功能，通过仪式感极强的交互设计与 AI 深度推理，为用户提供沉浸式、个性化的塔罗解读体验。

- **逻辑严丝合缝**：从洗牌算法到 AI 推理，每一环节有明确约束，拒绝随机性缺陷与 AI 幻觉。
- **仪式感优先**：每步交互模拟现实塔罗占卜的心理节奏，强化用户情感投入。
- **可扩展架构**：牌阵、牌意、AI Prompt 均采用配置化设计，支持快速迭代。
- **成本可控**：Token 消耗策略 + 缓存机制 + 模型降级，保障商业可持续性。

## 3. 技术架构决策（已确认）

> *以下五项 Open Questions 已经评审确认，本章为最终决策记录，研发可直接依此执行。*

### 3.1 AI 模型选型

**【决策】** MVP 阶段使用纯文本模型，不引入多模态图像识别。

| 项目 | 决策内容 |
| --- | --- |
| 首选模型 | DeepSeek-V2（国产合规，中文塔罗语料友好，成本低） |
| 备用模型 | GPT-3.5-Turbo（海外访问 fallback） |
| 为何不用多模态 | 牌面识别受拍照角度/光线影响大，且 Token 成本约为文本的 5-10 倍，MVP 不引入 |
| 单次占卜 Token 估算 | 系统提示词 + 牌阵上下文 + 输出 ≈ 800-1,500 Token（3张牌为例） |
| 输出长度限制 | 每条牌位解读 ≤ 200 Token，综合解读 ≤ 300 Token |

**Token 成本控制策略**

- 缓存常见框架：凯尔特十字等通用牌阵框架描述在本地拼接，减少系统提示词重复消耗。
- 每日免费次数限额：未登录 3 次/天，登录 10 次/天，超出按 0.01-0.02 元/次付费。
- 模型动态降级：高峰或高消耗请求时自动切换至更小版本模型。
- AI 接口统一封装：支持多模型热切换，不锁死单一供应商。

### 3.2 Session 持久化方案

**【决策】** Redis 存储活跃会话（TTL 24h + 活跃续期 2h），PostgreSQL 仅写入完成的占卜记录。

| 项目 | 方案 |
| --- | --- |
| 活跃会话存储 | Redis，数据结构为 JSON String，Key = session:{userId}:{sessionId} |
| 默认 TTL | 24 小时无操作后自动过期 |
| 活跃续期 | 每次 API 请求触发 Redis TTL 续期为 2h，防止用户操作中途丢失 |
| 完成记录落库 | PostgreSQL，仅存用户最终选牌+解读结果，中间步骤不落库 |
| 草稿会话上限 | 单用户最多 5 条未完成草稿（Redis），超出删除最早草稿 |
| 为何不纯用数据库 | 塔罗多步状态（选牌→抽牌→解读）需高频读写，Redis 性能更优，避免数据库写入压力 |

### 3.3 逆位默认开关

**【决策】** 逆位默认开启，首次进入弹窗说明，可在设置中随时关闭。

- **专业度保障**：逆位是标准塔罗的核心组成，默认关闭将损失解读丰富性。
- **技术零额外成本**：仅在选牌时增加正/逆位标识，无额外开发量。
- **新手引导**：首次进入时弹窗提示「逆位已开启，逆位并非厄运，而是能量的内化提醒」，降低心理抗拒。
- **视觉处理**：逆位牌边缘橙色微光 + 倒置标签，AI 解读中强调「阻塞或内化」而非「凶兆」。

### 3.4 历史记录存储量

**【决策】** 单用户保留 30 条完整占卜记录，写入时触发超出删除最早条目。

- **数据库字段**：user_id / record_id / spread_type / cards_json / interpretation_text / created_at
- **清理时机**：每次新记录写入时检查，若已有 30 条则删除 created_at 最早的一条（写入触发，非定时任务）。
- **草稿不计数**：未完成的占卜存于 Redis，不占用 30 条配额。
- **存储成本估算**：30 条 × 1KB ≈ 30KB/用户；百万用户约 30GB，完全可接受。

### 3.5 分享功能时间节点

**【决策】** v1.0 提供「保存为图片」，v1.1 增加微信海报分享（含小程序码）。

| 版本 | 功能 | 实现方式 |
| --- | --- | --- |
| v1.0 | 「保存为图片」本地分享 | Canvas 绘制牌面 + 解读摘要，前端生成，无需后端 |
| v1.1 | 「分享到微信」海报，含小程序码 | Canvas 生成图 + 后端提供用户昵称/牌阵数据接口 |
| v1.2 | 分享裂变统计、解锁奖励、自定义模板 | 后端埋点 + 活动系统 |

## 4. 核心交互：全流程状态机

### 4.1 状态流转

占卜流程分为五个严格顺序状态，任意状态均不可跳跃，后退需通过明确的「重新开始」操作。

- **STATE_IDLE** → 等待用户发起占卜
- **STATE_INTENT** → 用户输入问题意图（10-200 字）
- **STATE_SHUFFLE** → 洗牌动画（Fisher-Yates，毫秒级时间戳种子）
- **STATE_SELECT** → 用户选牌（扇形展开，防抖 300ms，同步锁定）
- **STATE_REVEAL** → 翻牌 + AI 流式解读

### 4.2 各阶段详细规格

| 阶段 | 后端/逻辑处理 | 前端表现 | 成功退出条件 | 潜在坑点 |
| --- | --- | --- | --- | --- |
| 1. 意图输入 | 敏感词过滤；长度校验 10-200 字；写入 Redis Draft | 沉浸式输入框，背景渐暗，实时字数显示 | 用户提交且通过校验 | 输入过短/无意义须前端引导提示 |
| 2. 洗牌 | 后端生成 78 张牌随机序列（Fisher-Yates）缓存至 Redis Session | 堆叠翻转动画，可点击加速跳过 | 动画结束或用户点击停止 | 动画掉帧破坏仪式感，限制同时渲染数 |
| 3. 选牌 | 锁定选中牌 ID 及正逆位，写入 Session；抽牌同步锁定 | 扇形展开，点击牌飞入对应牌位 | N 张牌全部选定（N 由牌阵决定） | 快速连续点击需防抖 300ms；并发锁定 |
| 4. 翻牌 | 按序触发单张牌释义加载；Session 标记已翻牌 ID | CSS 翻转动效，露出牌面原画 + 标题 | 当前牌阵所有牌翻完 | 翻牌顺序必须严格遵循牌阵定义 |
| 5. AI 解读 | 调用 AI 接口（结构化上下文注入）；超时 15s 触发降级 | 关键词先行动画 → 流式输出解读正文 | AI 响应完成或降级文案展示 | 超时白屏 >3s 不可接受，需进度反馈 |

## 5. 底层算法：随机性与发牌

### 5.1 洗牌算法规范

- **算法：** Fisher-Yates Shuffle（保证每张牌出现在每个位置的概率严格均等）
- **随机种子：** 用户点击「开始洗牌」按钮时的毫秒级时间戳 `Date.now()`，增强仪式参与感
- **禁止：** 直接使用 `Math.random()` 排序（统计分布不均匀）

### 5.2 正逆位规则

- **数据模型：** 每张牌含 `isReversed: Boolean`，洗牌时以 50% 概率独立随机赋值
- **全局开关：** 用户在设置中关闭逆位后，所有牌强制 `isReversed = false`

### 5.3 排重机制

单次牌阵抽取中，已选中的牌 ID 立即从候选池中原子性剔除，同一牌阵内不允许出现重复牌。
